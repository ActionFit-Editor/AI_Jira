# Jira Plan

Capture one explicitly requested title-only needs-plan intake, turn one new idea into an approved planned Jira todo issue, or refine one explicitly selected needs-plan issue under a transient planning lock that is used only for the approved write. Never implement or end a normal invocation in progress.

For title-only needs-plan intake, use this path only when the user explicitly requests it; never use it to bypass unresolved decisions in normal planning. Read repository guidance, query todo and progress separately, and obtain the user's decision before creating a likely duplicate. Prepare one exact Korean title without a managed description, decision questionnaire, full-plan approval view, branch, worktree, or implementation plan. Show the exact title and obtain explicit creation approval; a request that already names the exact title and directs creation satisfies approval, but an inferred follow-up does not. Then call only `python3 .claude/skills/jira-plan/scripts/ai_jira_write_cli.py create --summary "<Korean title>" --title-only-needs-plan`, never pass a description option, and never call Jira REST directly. Let the package command omit the description and enforce assignment, todo, active sprint, issue-type ID resolution, dry-run, and write gates. Re-read when available and require `descriptionContract.state=needs-plan`. Report the key and URL, leave the issue todo, and do not refine, transition, implement, commit, push, or open a PR.

1. Read `CLAUDE.md`, `AGENTS.md`, and linked repository guidance. Inspect relevant source and documentation without editing repository files.
2. When Jira is configured, query todo and progress separately through `.claude/skills/jira-plan/scripts/ai_jira_cli.py` and read details only to detect duplicates, dependencies, or overlap.
3. Read `references/planning-decision-collaboration.md`. Apply its convention precedence, one-to-three-question rounds, delegation lifetime, answer re-scan, and decision-closure confirmation while resolving the goal, scope, ordered implementation, completion conditions, validation, dependencies, risks, and migration implications. Do not silently select among material alternatives.
4. Only after every material decision is resolved and decision closure is explicitly confirmed, prepare the canonical Jira storage draft. Draft a Korean title. Keep `## QA 확인 필요 사항`, one `### 계획`, and QA content in Korean; write `## Auto Start`, `## Goal`, `## Scope`, `## Out of Scope`, `## Completion Criteria`, `## Validation Plan`, and `## Dependencies and Risks` plus their content in English. Require `Allowed`, `Prerequisites`, and `Decisions Required`, using `none` only when no unresolved decision remains. Give every English section content and use `None.` when no item applies. Keep technical identifiers unchanged. Record confirmed material choices and rationale under `## Scope` as `### Confirmed Decisions`.
5. Read `references/korean-approval-preview.md`, retain the exact canonical draft, show its complete Korean approval view without displaying the English storage body by default, and obtain explicit creation approval. Explain that approval writes the corresponding pre-preview canonical mixed-language draft. Decision closure, discussion, or partial approval is insufficient.
6. After approval, call only `python3 .claude/skills/jira-plan/scripts/ai_jira_write_cli.py create` with the exact pre-preview canonical title and a temporary UTF-8 `--description-file` outside the repository, then remove that file. Never back-translate the Korean approval view. Let the package-owned command enforce current-user assignment, todo status, sprint defaults, issue-type ID resolution, dry-run mode, and write gates.
7. Report the issue key and URL or the exact configuration/write blocker. Leave the issue in todo and do not implement or transition it.

For an explicitly selected needs-plan issue, capture its todo `updated` value and keep it in todo while preparing and retaining the complete canonical mixed-language draft. Follow `references/korean-approval-preview.md`, show its complete Korean approval view, and obtain explicit plan-update approval. Only in that approved continuation may you re-read and require the same todo status and captured `updated`; mismatch requires regeneration and reapproval without transition. After a match, transition to progress through `python3 .claude/skills/jira-plan/scripts/ai_jira_write_cli.py transition`, verify the state, and capture the post-transition `updated`. Call only the same locator's `update-description <ISSUE-KEY> --mode replace-plan --file <temp-file> --expected-updated <captured-value>` command with the pre-preview canonical draft, remove the file, re-read the contract, transition back to todo through the locator, and verify before responding. On update failure, attempt a verified todo rollback and report both failures if rollback fails. Never expire or steal a planning lock. Approval waiting, revisions, and uncertain canonical state remain todo; only an abrupt process or Jira failure may exceptionally strand progress.

If a likely duplicate exists, obtain the user's decision before creating another issue. Any requested revision invalidates the previous approval: update the canonical draft first, regenerate the full Korean view, and request approval again. Never create more than one issue, labels, epics, subtasks, Jira Advanced Roadmaps Plans, backlog exceptions, or bulk description migrations. Never create a worktree, edit repository files, implement, commit, push, open a PR, expose credentials, perform an unrestricted description overwrite, or call Jira REST directly.
